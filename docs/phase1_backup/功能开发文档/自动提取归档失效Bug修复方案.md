# 自动提取归档失效 Bug 修复方案

**版本**: Phase 1.5  
**状态**: ✅ 已完成  
**完成日期**: 2026-04-26  
**优先级**: 🔴 P0 (高优先级 - 修复数据丢失风险)  
**目标**: 修复自动触发提取时日志无法写入文件的问题，并优化归档内容的格式。

---

## 1. 问题背景 (Bug Report)

### 1.1 现象描述
*   **手动提取正常**：用户通过 `/archive` 指令或退出时触发的强制归档能成功写入 `D:\chat-log\log.txt`。
*   **自动提取失效**：当对话达到 5 轮触发自动提取时，终端显示“提取失败”或“保存成功”，但实际打开 `log.txt` 文件，发现并没有新增任何内容。
*   **日志格式混乱**：终端打印的自动提取内容包含大量工具调用的原始 JSON 和冗长的格式化 Prompt，视觉效果极差（“像屎一样”）。

### 1.2 根本原因分析
通过代码审查 (`chat_compress_client.py`)，发现以下逻辑断层：
1.  **方法缺失/失效**：在 `_extract_5w_info` 方法的 L1244 行，调用了 `_save_failed_extraction`。该方法在当前代码中可能不存在，或者其内部仅执行了 `print` 而未执行真正的文件 I/O 操作。
2.  **路径不统一**：手动提取走的是健壮的 `_save_short_consultation_record` 路径，而自动提取走的是一个未完成的“半成品”路径。
3.  **内容冗余**：自动提取在构建日志时，直接将包含 System Prompt 模板和 Tool 返回值的原始消息列表存入，导致日志文件膨胀且难以阅读。

---

## 2. 实施方案

### 2.1 核心思路
1.  **统一归档接口**：废除无效的 `_save_failed_extraction`，在自动提取失败时，统一调用已验证过的 `_save_short_consultation_record` 进行兜底归档。
2.  **精简日志内容**：在写入文件前，对 `recent_messages` 进行清洗，剔除冗长的 System Prompt 模板和工具调用的原始 JSON 返回值。
